## 1. The Investigation
My role was to analyze the artifacts left behind after the reported "security incident." We had two distinct sets of data to review: the "Noisy" attack (Simulation 1) and the "Stealth" attack (Simulation 2).

### 1.1 Preliminary Findings
Upon reviewing the file system of the victim machine (`192.168.0.81`), initial scans using standard File Integrity tools (Wazuh dashboard) showed **zero critical alerts** during the time of the Stealth attack.
*   **Observation**: No "File Deleted" alerts were triggered.
*   **Observation**: No "File Modified" alerts were triggered for the `~/Music` directory target.

Use of the `stat` command on files in the target directory showed "Last Modified" dates from the year **2010**. This was immediately suspicious, as the operating system itself was installed in 2024. This timestamp inconsistency (Temporal Anomaly) was the first indicator of **Timestomping**.

### 2.2 Comparative Analysis: Why Traditional Tools Failed
The most critical finding of this investigation is the failure of the File Integrity Monitor (Wazuh).

**The Failure Mechanism (Wazuh):**
1.  **Scanning Frequency (The Time Gap)**: Wazuh's `syscheck` module operates on a scheduled polling interval (default: every 12 hours). It scans the file system, creates a checksum hash of files, and compares them to the previous scan.
2.  **Attack Speed**: The "Stealth Agent" created the droplet, executed it, and wiped it **within 300 milliseconds**.
3.  **Result**: Because the file existed and was deleted *between* two scans, Wazuh never hashed it. To the FIM, the file never "existed."

**The Success Mechanism (Auditd):**
1.  **Kernel Interception**: Unlike Wazuh, `auditd` does not poll. It hooks directly into the Linux Kernel (using the `kaudit` framework).
2.  **Real-Time Logging**: Every time a program requests a system call (like `unlink` to delete or `utime` to change dates), the kernel pauses, logs the event, and then allows the action.
3.  **Result**: Even though the file existed for less than a second, the *action* of deleting it was intercepted and logged permanently.

| Attack Mode | Wazuh (FIM) Detection | Auditd (Kernel) Detection | Evidence Left |
| :--- | :--- | :--- | :--- |
| **Noisy** | **Detected** (File Deleted) | **Detected** (Unlink syscall) | Standard file fragments (Recoverable) |
| **Stealth** | **MISSED** (Due to Time Gap) | **Detected** (Rename + Truncate + Unlink) | Wiped data + Backdated Metadata (Irrecoverable) |

## 3. Conclusion & Recommendations
The investigation confirms that the attacker successfully bypassed standard periodic monitoring by exploiting the "Time Gap"—executing their actions faster than the FIM polling interval.

**Recommendations:**
1.  **Shift to Real-Time**: Organizations must move from periodic FIM (like standard Wazuh config) to real-time kernel monitoring (Auditd or eBPF-based EDR sensors) for high-value directories.
2.  **Monitor Metadata**: Security rules should alert on "Retro-dating" (setting timestamps >1 year in the past), which is a high-fidelity indicator of Timestomping.
3.  **Wiping Detection**: Alert on rapid, sequential "Rename" + "Delete" operations, which is the signature of secure wiping tools.
